     524              : // The ~uvm_callbacks~ class provides a base class for implementing callbacks,
     525              : // which are typically used to modify or augment component behavior without
     526              : // changing the component class. To work effectively, the developer of the
     527              : // component class defines a set of "hook" methods that enable users to
     528              : // customize certain behaviors of the component in a manner that is controlled
     529              : // by the component developer. The integrity of the component's overall behavior
     530              : // is intact, while still allowing certain customizable actions by the user.
     531              : // 
     532              : // To enable compile-time type-safety, the class is parameterized on both the
     533              : // user-defined callback interface implementation as well as the object type
     534              : // associated with the callback. The object type-callback type pair are
     535              : // associated together using the <`uvm_register_cb> macro to define
     536              : // a valid pairing; valid pairings are checked when a user attempts to add
     537              : // a callback to an object.
     538              : //
     539              : // To provide the most flexibility for end-user customization and reuse, it
     540              : // is recommended that the component developer also define a corresponding set
     541              : // of virtual method hooks in the component itself. This affords users the ability
     542              : // to customize via inheritance/factory overrides as well as callback object
     543              : // registration. The implementation of each virtual method would provide the
     544              : // default traversal algorithm for the particular callback being called. Being
     545              : // virtual, users can define subtypes that override the default algorithm,
     546              : // perform tasks before and/or after calling super.<method> to execute any
     547              : // registered callbacks, or to not call the base implementation, effectively
     548              : // disabling that particular hook. A demonstration of this methodology is
     549              : // provided in an example included in the kit.

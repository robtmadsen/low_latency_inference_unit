      37              : // Topic: Intro
      38              : //
      39              : // A resource is a parameterized container that holds arbitrary data.
      40              : // Resources can be used to configure components, supply data to
      41              : // sequences, or enable sharing of information across disparate parts of
      42              : // a testbench.  They are stored using scoping information so their
      43              : // visibility can be constrained to certain parts of the testbench.
      44              : // Resource containers can hold any type of data, constrained only by
      45              : // the data types available in SystemVerilog.  Resources can contain
      46              : // scalar objects, class handles, queues, lists, or even virtual
      47              : // interfaces.
      48              : //
      49              : // Resources are stored in a resource database so that each resource can
      50              : // be retrieved by name or by type. The database has both a name table
      51              : // and a type table and each resource is entered into both. The database
      52              : // is globally accessible.
      53              : //
      54              : // Each resource has a set of scopes over which it is visible.  The set
      55              : // of scopes is represented as a regular expression.  When a resource is
      56              : // looked up the scope of the entity doing the looking up is supplied to
      57              : // the lookup function.  This is called the ~current scope~.  If the
      58              : // current scope is in the set of scopes over which a resource is
      59              : // visible then the resource can be retuned in the lookup.
      60              : //
      61              : // Resources can be looked up by name or by type. To support type lookup
      62              : // each resource has a static type handle that uniquely identifies the
      63              : // type of each specialized resource container.
      64              : //
      65              : // Multiple resources that have the same name are stored in a queue.
      66              : // Each resource is pushed into a queue with the first one at the front
      67              : // of the queue and each subsequent one behind it.  The same happens for
      68              : // multiple resources that have the same type.  The resource queues are
      69              : // searched front to back, so those placed earlier in the queue have
      70              : // precedence over those placed later.
      71              : //
      72              : // The precedence of resources with the same name or same type can be
      73              : // altered.  One way is to set the ~precedence~ member of the resource
      74              : // container to any arbitrary value.  The search algorithm will return
      75              : // the resource with the highest precedence.  In the case where there
      76              : // are multiple resources that match the search criteria and have the
      77              : // same (highest) precedence, the earliest one located in the queue will
      78              : // be one returned.  Another way to change the precedence is to use the
      79              : // set_priority function to move a resource to either the front or back
      80              : // of the queue.
      81              : //
      82              : // The classes defined here form the low level layer of the resource
      83              : // database.  The classes include the resource container and the database
      84              : // that holds the containers.  The following set of classes are defined
      85              : // here:
      86              : //
      87              : // <uvm_resource_types>: A class without methods or members, only
      88              : // typedefs and enums. These types and enums are used throughout the
      89              : // resources facility.  Putting the types in a class keeps them confined
      90              : // to a specific name space.
      91              : //
      92              : // <uvm_resource_options>: policy class for setting options, such
      93              : // as auditing, which effect resources.
      94              : //
      95              : // <uvm_resource_base>: the base (untyped) resource class living in the
      96              : // resource database.  This class includes the interface for setting a
